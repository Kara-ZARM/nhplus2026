## 1. DAO-Pattern: create-Methode

> Erläutert das Zusammenspiel von `DaoImp`, `PatientDao` und der Datenbanktabelle `patient`, wenn die im Interface `Dao` definierte Methode `void create(T t) throws SQLException` aufgerufen wird.

Die Dao Klasse ist ein Interface in der die Methode "void create (T t)" existiert, wobei "(T t)" ein Platzhalter für eine beliebige Klasse darstellt, da das Programm nicht nur mit Patienten arbeitet, sondern auch mit z.B. "Treatments" und somit erweiterbar bleibt. DaoImp implementiert Dao und überschreibt die genannte Methode, die mit dem Datentyp der Klasse "Patient" aufgerufen wird und in der die abstrakte Methode "protected abstract PreparedStatement getCreateStatement(T t)" aufgerufen wird, die in PatientDao geerbt und implementiert wird. In PatientDao wird in dieser Methode ein SQL Statement als String vorbereitet, mit dem ein preparedStatement erstellt wird und dann mithilfe von Gettern die benötigten Daten für die Tabelle in das preparedStatement gesetzt und zurückgegeben werden. Danach wird statement.executeUpdate() in DaoImp ausgeführt, mit der das SQL Statement ausgeführt und ein Eintrag in der Patient Datenbank erstellt wird.

## 2. Erweiterung um ein Pfleger-Modul

> Was müsstet ihr tun, um ein Modul zur Verwaltung von Pflegekräften zu ergänzen? Welchen Vorteil bietet die Architektur mit generischem `DaoImp<T>` gegenüber einer Architektur mit konkreten DAO-Klassen pro Model?

- Caregiver Datenbank Tabelle erstellen
- Caregiver Klasse (in model) erstellen
- CaregiverDao Klasse (in datastorage) erstellen
- DaoFactory um createCaregiverDao() erweitern
- AllCaregiverController (in controller) erstellen
- AllCaregiverView.fxml erstellen

Der Vorteil der Architektur mit generischem DaoImp<T> ist der, dass man die Klassen als Datentyp flexibler übergeben kann und somit das Interface bzw. dessen Methoden für alle Klassen, die mit der Datenbank zu tun haben, anwenden kann.
Dadurch wird Code-Duplizierung vermieden, die Klassen werden kleiner gehalten und die Klassen werden entkoppelt.

## 3. ConnectionBuilder, DaoFactory und das Singleton-Pattern

> Erklärt die Aufgaben der Klassen `ConnectionBuilder` und `DaoFactory`. Warum hält `PatientDao` keine eigene Datenbankverbindung? Warum sind beide Klassen Singletons?

ConnectionBuilder hat allein die Aufgabe eine Verbindung zur Datenbank herzustellen.
DaoFactory ist dafür verantwortlich Dao Classes zu erstellen, überlässt aber den Subclasses welcher spezifischer Typ sie sind (Patient, Treatment, Caregiver)
PatientDao hat keine eigene Datenbankverbindung, weil davon nur eine existieren soll und weil es so auch eine klare Trennung von Zuständigkeiten gibt.
Beide Klassen sind Singletons, damit es nur einen Ursprung gibt von wo aus Verbindungen (DB) oder Klassen(Factory) hergestellt werden können.
Mehrere parallele Datenbank-Connections könnten zu Inkonsistenzen führen, da ohne klare Regeln auf eine Ressource zugegriffen werden könnte. Daraus könnten schwer zu debuggende Race Conditions entstehen.


## 4. MVP-Pattern

> Erklärt das Zusammenspiel von `Patient`, `AllPatientController` und `AllPatientView.fxml`, wenn ein Benutzer einen neuen Patienten anlegt und auf *Hinzufügen* klickt. Worin liegt der Unterschied zu MVC?

Über den Button "Hinzufügen" bzw. dessen fxid "buttonAdd" wird die Methode "handleAdd" in AllPatientController aufgerufen.
In der Methode werden Variablen initialisiert, die ihre Daten aus den korrespondierenden Textfeldern (über deren xfids) per Getter beziehen.
Mit den Daten aus den Variablen wird ein neues Model Objekt der Klasse Patient erzeugt und anschließend wird über die Dao Methode "create" ein neuer Eintrag in der Datenbank erstellt (siehe Aufgabe 1).
Danach wird die Tabelle in View über readAllAndShowInTableView() aktualisiert und die Textfelder zurückgesetzt.

Der Unterschied liegt darin, dass bei dem MVP-Pattern Änderungen im Model zum Presenter weitergegeben werden und von dort der View geupdated wird, und andersherum.
Bei der MVC kommuniziert der User mit einem Controller, der das Model updated und das Model die View updated.

## 5. Erste eigene Änderung

> Welchen Weg nehmen die Daten vom `SetUpDB`-Aufruf bis zur Anzeige in der GUI?

SetUpDB ruft auf:
- wipeDb(connection); -> löscht alle Tabellen und damit gesamten Inhalt der Datenbank
- setUpTablePatient(connection); -> erstellt neue Tabelle für Patienten Informationen
- setUpTableTreatment(connection); -> erstellt neue Tabelle für Treatment Informationen
- setUpPatients(); -> füllt Tabelle Patient mit Patienten Informationen
- setUpTreatments(); -> füllt Tabelle Treatment mit Treatment Informationen

Controller erstellen Elemente, die in der GUI angezeigt werden sollen, mit entsprechenden Methoden(bspw.: handleOnEditFirstname()) werden den Elementen die Daten aus den Attributen der model Classes zugewiesen.
TextField mit entsprechender fx:id (Attribut aus Controller) zeigt Daten in GUI an

- neuer Patient erscheint in Datenbank und wird angezeigt
